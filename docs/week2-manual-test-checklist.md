# Week 2 Manual Test Checklist

이 문서는 `Next.js + Supabase` 기준 2주차 핵심 흐름을 수동으로 확인할 때 사용하는 체크리스트입니다.

## 테스트 전 준비

1. 프론트 개발 서버 실행
2. Supabase 환경 변수가 올바르게 설정되어 있는지 확인
3. 브라우저에서 `http://localhost:3000` 접속
4. 이전 로그인 세션이 남아 있으면 상황에 따라 로그아웃 후 다시 시작

## 1. 인증 / 현재 사용자 확인

### 목표

- 학교 이메일 사용자만 앱 흐름에 진입할 수 있는지 확인
- 로그인 후 현재 사용자 식별이 되는지 확인

### 확인 항목

- [ ] 메인 화면 또는 헤더에서 `/auth`로 이동 가능하다.
- [ ] 학교 이메일(`kmu.ac.kr`)이 아닌 계정으로는 bootstrap이 차단된다.
- [ ] 학교 이메일 계정으로 회원가입 또는 로그인할 수 있다.
- [ ] 로그인 후 헤더 상태가 바뀐다.
- [ ] `GET /api/auth/me` 응답이 정상이다.

### 기대 결과

- 로그인 후 현재 앱 사용자 정보가 존재한다.
- `users.auth_user_id` 기준으로 사용자가 식별된다.

## 2. 온보딩 / 프로필 저장

### 목표

- 단계형 온보딩 화면에서 프로필 정보를 저장할 수 있는지 확인

### 확인 항목

- [ ] `/onboarding` 진입 시 로그인되지 않았다면 로그인 유도 화면이 보인다.
- [ ] 로그인 후 온보딩 1~5단계가 정상적으로 보인다.
- [ ] 이름, 캠퍼스, 학과, 학년이 입력된다.
- [ ] 역할 태그와 툴 정보가 입력된다.
- [ ] 포트폴리오 링크와 작업물 내 역할을 입력할 수 있다.
- [ ] 협업 가능 상태, 협업 유형, 주당 가능 시간 입력이 가능하다.
- [ ] 마지막 단계에서 저장 후 `/projects`로 이동한다.
- [ ] 새로고침 후에도 저장한 프로필이 다시 불러와진다.

### 관련 API

- `POST /api/auth/bootstrap`
- `GET /api/profiles/me`
- `POST /api/profiles`
- `PUT /api/profiles/me`

## 3. 포트폴리오 등록 / 조회

### 목표

- 내 포트폴리오 저장 및 조회가 되는지 확인

### 확인 항목

- [ ] 온보딩 단계 또는 포트폴리오 등록 화면에서 포트폴리오를 저장할 수 있다.
- [ ] 외부 링크 형식이 잘못되면 저장이 차단된다.
- [ ] 저장 후 내 포트폴리오 목록 조회가 가능하다.
- [ ] 특정 프로필 포트폴리오 조회가 가능하다.

### 관련 API

- `GET /api/portfolios/me`
- `POST /api/portfolios`
- `GET /api/profiles/{profileId}/portfolios`

## 4. 프로젝트 목록 / 상세 / 등록

### 목표

- 프로젝트를 목록에서 보고, 상세 진입하고, 새로 등록할 수 있는지 확인

### 확인 항목

- [ ] `/projects`에서 프로젝트 목록이 보인다.
- [ ] 검색어 필터가 동작한다.
- [ ] 캠퍼스 필터가 동작한다.
- [ ] 역할 필터가 동작한다.
- [ ] 상태 필터가 동작한다.
- [ ] 프로젝트 상세(`/projects/{id}`) 진입이 가능하다.
- [ ] 로그인한 상태에서 프로젝트 등록이 가능하다.
- [ ] 로그인하지 않은 상태에서는 등록 액션이 제한된다.

### 관련 API

- `GET /api/projects`
- `GET /api/projects/{projectId}`
- `POST /api/projects`
- `PUT /api/projects/{projectId}`

## 5. 지원 / 제안 / 현황

### 목표

- 프로젝트 지원과 인재 제안이 생성되고 현황 화면에 표시되는지 확인

### 확인 항목

- [ ] 프로젝트 상세에서 지원이 가능하다.
- [ ] 자기 프로젝트에는 지원할 수 없다.
- [ ] 같은 프로젝트에 중복 지원이 막힌다.
- [ ] 추천 인재 영역에서 제안하기가 가능하다.
- [ ] 자기 자신에게는 제안할 수 없다.
- [ ] 같은 프로젝트로 같은 사용자에게 중복 제안이 막힌다.
- [ ] `/applications`에서 내가 보낸 지원/제안이 보인다.
- [ ] `/applications`에서 내가 받은 지원/제안이 보인다.
- [ ] 받은 지원/제안에 대해 수락/거절이 가능하다.

### 관련 API

- `POST /api/applications`
- `GET /api/applications/me`
- `GET /api/applications/received`
- `PATCH /api/applications/{applicationId}`
- `POST /api/proposals`
- `GET /api/proposals/me`
- `PATCH /api/proposals/{proposalId}`

## 6. 추천 인재

### 목표

- 프로젝트 화면에서 추천 인재가 실제 데이터로 보이는지 확인

### 확인 항목

- [ ] `/projects` 화면의 추천 인재 카드가 보인다.
- [ ] 추천 인재 카드 정보가 mock 고정값이 아니라 API 응답 기준으로 보인다.

### 관련 API

- `GET /api/recommendations/talents`

## 7. 비정상 케이스

### 확인 항목

- [ ] 로그인하지 않은 상태에서 보호된 데이터 요청 시 적절한 실패 응답이 나온다.
- [ ] 권한 없는 프로젝트 수정이 차단된다.
- [ ] 권한 없는 프로필 수정이 차단된다.
- [ ] 잘못된 입력값은 저장되지 않는다.
- [ ] 에러 응답에 DB 경로, 스택트레이스 같은 민감 정보가 노출되지 않는다.

## 최종 완료 기준

- [ ] 인증
- [ ] 온보딩
- [ ] 포트폴리오
- [ ] 프로젝트
- [ ] 지원/제안
- [ ] 추천 인재
- [ ] 비정상 케이스

위 항목을 모두 통과하면 2주차 핵심 흐름은 프론트 연결 기준으로 수동 검증 완료로 본다.
