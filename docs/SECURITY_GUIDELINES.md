# Campus Link 보안 지침

Campus Link는 학교 기반 협업 매칭 웹서비스이므로, 일반 웹 보안뿐 아니라 학생 정보와 포트폴리오 자료 보호를 함께 고려해야 합니다. MVP 단계에서는 아래 항목을 필수 보안 기준으로 둡니다.

## 1. 인증 및 로그인

- 비밀번호는 평문으로 저장하지 않고 안전한 해시 방식으로 저장합니다.
- 학교 이메일 인증을 통해 외부 사용자의 무분별한 가입을 제한합니다.
- 로그인 실패 횟수 제한을 적용해 무차별 대입 공격을 방지합니다.
- 비밀번호 재설정 링크는 짧은 만료 시간과 1회성 토큰을 사용합니다.
- 관리자 계정에는 가능하면 MFA를 적용합니다.

## 2. 권한 관리

- 사용자는 본인 프로필, 본인 지원 내역, 본인이 등록한 프로젝트만 수정할 수 있어야 합니다.
- 프로젝트 수정, 지원 상태 변경, 제안 수락/거절은 서버에서 권한을 검증합니다.
- URL의 사용자 ID나 프로젝트 ID를 바꿔 다른 사용자의 정보에 접근할 수 없도록 합니다.
- 관리자 기능은 별도 권한을 가진 계정만 접근 가능하도록 제한합니다.

### 2.1 PostgreSQL RLS (MVP P0)

MVP 공개 테이블(`users`, `profiles`, `portfolio_items`, `projects`, `applications`)은 모두 RLS가 활성화되어 있으며, 명시적 정책이 없으면 기본 거부(default-deny)입니다. `using (true)` 같은 전면 허용 정책은 사용하지 않습니다.

- **신원 매핑:** `public.current_app_user_id()`가 `auth.uid()`를 bigint `public.users.id`로 변환합니다. `getCurrentAppUser()`가 `users`를 조회할 수 있도록, 인증 사용자는 `auth_user_id = auth.uid()`인 본인 행만 `SELECT`할 수 있습니다.
- **profiles:** 인증 사용자는 프로필을 읽을 수 있고, `INSERT`/`UPDATE`는 `user_id = current_app_user_id()`인 본인 행만 가능합니다.
- **portfolio_items:** 인증 사용자는 포트폴리오를 읽을 수 있고, `INSERT`/`UPDATE`/`DELETE`는 소유자만 가능합니다.
- **projects:** 인증 사용자는 `RECRUITING` 프로젝트와 본인 소유 프로젝트를 `SELECT`할 수 있고, `INSERT`/`UPDATE`/`DELETE`는 소유자만 가능합니다.
- **applications:** 지원자는 `PENDING` 상태로 본인 지원을 생성·조회할 수 있습니다. 프로젝트 소유자는 해당 프로젝트 지원을 조회할 수 있습니다. 상태 전환(`PENDING`→`ACCEPTED`/`REJECTED`, `PENDING`→`CANCELED`)은 `owner_decide_application()` 및 `applicant_withdraw_application()` 보안 함수로만 수행합니다. 테이블에 직접 `UPDATE` 정책은 두지 않습니다.
- **상태 값 검증:** `applications.application_status`, `projects.recruitment_status`, `profiles.collaboration_status`에 `CHECK` 제약이 있습니다.

RLS 검증 스크립트: `frontend/supabase/tests/rls-p0.sql`  
마이그레이션 적용 후 실행:

```bash
cd frontend
npx supabase db push
psql "$SUPABASE_DB_URL" -f supabase/tests/rls-p0.sql
```

서버 코드는 세션 클라이언트로 RLS를 통과시키고, 서비스 롤 클라이언트는 부트스트랩·유지보수 등 명시적으로 문서화된 경우에만 사용합니다.

## 3. 개인정보 보호

- 이름, 닉네임, 학교 이메일, 캠퍼스, 학과, 학년 등 개인정보는 서비스 운영에 필요한 범위만 수집합니다.
- 학번, 전화번호, 학생증 이미지처럼 민감도가 높은 정보는 MVP에서 기본 수집하지 않습니다.
- 학생증 또는 포털 캡처 인증을 확장 기능으로 도입할 경우, 민감정보 가림 안내와 인증 후 이미지 삭제 정책을 포함합니다.
- 로그에는 비밀번호, 인증 토큰, 민감 개인정보가 남지 않도록 합니다.
- 사용자가 탈퇴하면 계정 정보와 불필요한 개인정보를 삭제하거나 익명화합니다.

## 4. 입력값 검증

- 회원가입, 로그인, 프로필, 프로젝트 등록, 포트폴리오 등록, 댓글/메시지, 검색어 등 모든 입력값을 서버에서 검증합니다.
- SQL Injection 방지를 위해 문자열 조합 쿼리 대신 파라미터 바인딩 또는 ORM을 사용합니다.
- XSS 방지를 위해 사용자가 입력한 텍스트를 화면에 출력할 때 이스케이프 처리합니다.
- 클라이언트 검증은 사용자 편의를 위한 보조 수단으로만 사용하고, 최종 검증은 서버에서 수행합니다.

## 5. 세션 및 토큰 관리

- 인증 쿠키를 사용할 경우 `HttpOnly`, `Secure`, `SameSite` 옵션을 적용합니다.
- JWT 또는 세션 토큰에는 만료 시간을 설정합니다.
- 로그아웃 시 세션 또는 토큰을 무효화합니다.
- 장시간 미사용 시 자동 로그아웃 또는 재인증을 요구합니다.

## 6. 파일 업로드 보안

- 포트폴리오 이미지, 영상, GIF 업로드는 허용된 확장자와 MIME 타입만 받습니다.
- 파일 크기 제한을 적용합니다.
- 업로드 파일명은 서버에서 새로 생성해 저장합니다.
- 실행 가능한 파일 업로드를 금지합니다.
- 파일은 공개 접근 범위를 최소화하고, 필요한 경우 서명 URL로 접근을 제어합니다.

## 7. API 보안

- 인증이 필요한 API는 모든 요청에서 인증 토큰을 검증합니다.
- API별로 사용자의 권한을 확인합니다.
- 반복 요청을 제한하기 위해 Rate Limit을 적용합니다.
- 에러 메시지에 서버 경로, DB 정보, 스택트레이스가 노출되지 않도록 합니다.
- CORS는 프론트엔드 배포 도메인 등 필요한 출처만 허용합니다.

## 8. HTTPS 및 보안 헤더

- 운영 환경의 모든 페이지와 API는 HTTPS만 허용합니다.
- HSTS를 적용해 HTTPS 사용을 강제합니다.
- Content-Security-Policy를 적용해 XSS 피해 범위를 줄입니다.
- `X-Frame-Options` 또는 CSP `frame-ancestors`를 사용해 클릭재킹을 방지합니다.

## 9. 의존성 및 배포 보안

- Next.js, Spring Boot, Supabase 관련 패키지를 정기적으로 업데이트합니다.
- GitHub Dependabot 또는 유사한 도구로 취약한 의존성을 점검합니다.
- `.env`, API Key, DB 비밀번호, Supabase Service Role Key는 Git에 커밋하지 않습니다.
- 개발 환경과 운영 환경의 환경변수와 데이터베이스를 분리합니다.
- Supabase Storage와 DB 권한 정책은 최소 권한 원칙에 맞게 설정합니다.

## 10. 로그 및 모니터링

- 로그인 실패, 권한 오류, 프로젝트 수정, 지원 상태 변경, 관리자 작업은 로그로 남깁니다.
- 비정상적으로 많은 요청이나 반복 실패 요청을 감지할 수 있도록 합니다.
- 보안 사고 발생 시 계정 비활성화, 토큰 무효화, 관련 로그 확인 절차를 마련합니다.
- 로그에는 민감 개인정보가 포함되지 않도록 합니다.

## 11. 관리자 보안

- 관리자 계정은 최소 인원에게만 부여합니다.
- 관리자 작업은 누가, 언제, 무엇을 변경했는지 기록합니다.
- 사용자 삭제, 인증 승인, 신고 처리 같은 중요한 작업은 재확인 절차를 둡니다.
- 기본 관리자 계정과 기본 비밀번호는 사용하지 않습니다.

## 12. 보안 점검 기준

- 배포 전 OWASP Top 10 기준으로 접근 제어, 인증, 입력값 검증, 보안 설정을 점검합니다.
- 주요 기능 변경 시 인증/권한 테스트를 다시 수행합니다.
- 파일 업로드, 포트폴리오 링크, 프로젝트 등록 기능은 악성 입력과 권한 우회 가능성을 집중 점검합니다.
- 참고 기준: [OWASP Top 10](https://owasp.org/Top10/), [OWASP Proactive Controls](https://owasp.org/www-project-proactive-controls/)

## MVP 필수 적용 항목

- 학교 이메일 인증
- 사용자별 프로필/프로젝트/지원 내역 권한 검증
- 비밀번호 해시 저장
- 서버 측 입력값 검증
- HTTPS 사용
- 파일 업로드 확장자와 용량 제한
- 환경변수와 API Key Git 커밋 금지
- 개인정보와 인증 토큰 로그 기록 금지
